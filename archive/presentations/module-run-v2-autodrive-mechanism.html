Module Run v2 · Codex Autodrive Operating Model

从“定时醒来”到“安全推进”的无人值守自动执行机制

当前机制不是让 Codex 随机开工，而是让它在 Codex 自动化定时唤醒后，先读取仓库内的持久状态，再经过 startup、lease、scope、thread、closeout、hook、evidence 等多层门禁，只有在边界清晰且验证可落地时才继续推进。 遇到硬阻断、活跃线程、脏 worktree、高风险面或缺证据时，它必须停下并交接。

ACTIVE Codex automation id: tiku-module-run-v2-autopilot
hourly 项目状态记录的自动化唤醒节奏
worktree 自动化执行环境，配合 lease 避免互相干扰
guarded 只允许本地安全推进，高风险动作保持 blocked
01

总体结论

自动驾驶机制已经形成闭环，但它的成熟形态是“受控无人值守”，不是无边界自治。

能自动做什么

定时醒来、恢复状态、判断是否已有运行占用通道、选择或准备合法 pending 任务、生成 handoff、执行低风险本地任务、跑门禁、写 evidence、在允许范围内继续。

什么时候必须停

远端领先、脏 worktree、active lease、缺 task plan/evidence、高风险面、scope 漂移、验证失败、需要新线程但 launch policy 不允许、Cost Calibration Gate 等场景必须停。

怎么保证不各干各的

通过 automation lease、startup readiness、task-queue 状态、thread rollover、handoff 文件、Git inventory 和 stopped automation hygiene 共同协调。

02

机制架构图

Codex 自动化只是入口；真正的控制逻辑落在仓库内的 YAML、SOP、PowerShell 脚本、Git Hooks 与 evidence 中。

03

状态源：自动化不是靠聊天记忆推进

每次醒来都必须从仓库里的持久文件恢复，而不是凭上一段对话继续。

状态源 职责 自动化如何使用
AGENTS.md 项目命名规范、Git 纪律、AI 执行纪律、禁区和交付 checklist。 任何修改前必须读取；决定术语、分支、提交、验证和最终自检规则。
docs/03-standards/code-taste-ten-commandments.md 代码品味与交付底线。 修改前读取，结束时输出品味合规自检。
docs/02-architecture/adr/ 架构决策：技术栈、运行分层、多端契约、环境隔离、staging 边界。 防止 Agent 越过既定架构，例如不直接返回 DB row、不绕过服务层、不触碰 staging/prod。
project-state.yaml 当前阶段、当前任务、自动化模式、Codex automation id、lease 路径、handoff。 startup gate 的主要恢复点，用来判断当前任务、下一步和自动化批准状态。
task-queue.yaml 任务队列、依赖、allowedFiles、blockedFiles、riskTypes、validationCommands。 任务领取、scope scan、pre-work、pre-edit、pre-commit 的判断依据。
advanced-edition-domain-module-run-matrix.yaml Module Run v2 的 execution module、localFullLoopGate、threadRolloverGate、automationHandoffPolicy。 决定模块推进顺序、批次边界、线程切换、下一模块候选和 blocked remainder。
04

自动化启动流程

Codex 自动化醒来后的第一目标不是开工，而是判断“现在是否应该开工”。

定时醒来 Codex app automation 按项目状态记录的 hourly 节奏运行。
读状态源 读取 state、queue、matrix、latest evidence。
Startup Gate Test-ModuleRunV2AutomationStartupReadiness.ps1
Lease Gate Test-ModuleRunV2AutomationLeaseReadiness.ps1
Worktree Hygiene 发现 stale/dirty automation worktree 时停止。
Queue Decision continue_current_task / prepare_next_task / closeout_recovery。
Autopilot Invoke-ModuleRunV2Autopilot.ps1
Hard Stop active lease、无合法任务、blocked gate、远端漂移则停。
continue_current_task当前 task 为 in_progress，且启动门禁通过。
prepare_next_task当前 task 已 done，队列里有 pending 且可准备。
closeout_recovery当前 task 完成但需要恢复 closeout 对账。
stop_existing_run_active已有 non-expired lease，说明另一个运行拥有通道。
stop_for_hard_block启动门禁发现不可自动修复的问题。
no_executable_task没有合法 pending/in-progress 任务。
05

任务执行机制

执行不是“看到 pending 就改代码”，而是先过 pre-work/pre-edit，再按 Batch 或 task 的 focused scope 推进。

执行前硬门禁

1
pre-workTest-ModuleRunV2WorkReadiness.ps1 -Mode pre-work -TaskId <task> 校验任务存在、未完成、路径齐全、分支合法。
2
pre-edit同脚本的 -Mode pre-edit 校验 planned files 是否都在 allowedFiles 且不碰 blockedFiles。
3
unattendedTest-ModuleRunV2UnattendedReadiness.ps1 判断是否可继续无人值守。
4
implementation auto-seedTest-ModuleRunV2ImplementationAutoSeedReadiness.ps1 控制 planning 后是否可进入低风险实现。

执行中约束

allowedFiles 唯一允许触碰的文件面。
blockedFiles 只要触碰就硬阻断，除非任务有新的明确批准。
riskTypes 决定是否需要人工批准或额外 evidence。
validationCommands 完成前必须运行并记录的验证命令。
taskKind 区分 docs_only、implementation、local_verification、closeout、blocked_gate 等执行性质。
06

线程控制与 handoff

“新线程”不是普通脚本能直接创建的动作，而是 PowerShell 产出决策，Codex agent layer 再调用线程工具。

决策脚本

Test-ModuleRunV2ThreadRolloverReadiness.ps1

根据 Batch 数、上下文压力、高风险边界和 closeout 状态给出 continue/suggest/require new thread。

交接生成

New-ModuleRunV2ThreadHandoff.ps1

生成包含 current module、completed batches、latest SHA、evidence、blocked gates、next allowed task 的 handoff。

启动策略

Test-ModuleRunV2ThreadLaunchPolicy.ps1

只有返回 launch_new_thread，且 Codex 线程工具可用时，agent layer 才可调用 create_thread。

关键边界

PowerShell 脚本不直接创建 Codex 线程；它只输出机器可读的决策。真正的线程创建由 Codex 的 create_thread / send_message_to_thread 工具执行，并且必须满足 launch policy。

07

Hooks 与门禁实现

Hook 层保证“人或 Agent 想跳过流程”时，本地 Git 也会拦住关键风险。

层级 实现文件 强度 检查内容
pre-work / pre-edit Test-ModuleRunV2WorkReadiness.ps1 hard block TaskId、计划/evidence/audit 路径、分支、allowedFiles、blockedFiles。
pre-commit .husky/pre-commit
Test-ModuleRunV2PreCommitHardening.ps1		 hard block scope scan、敏感 evidence、术语禁用、lint-staged、lint、typecheck。
post-commit Test-ModuleRunV2PostCommitReadiness.ps1 advisory last commit、changed files、task/evidence/audit inventory、scope summary。
pre-push .husky/pre-push
Test-ModuleRunV2PrePushReadiness.ps1		 hard block GitCompletionReadiness、远端领先、state SHA 祖先关系、evidence/audit 存在。
module-closeout Test-ModuleRunV2ModuleCloseoutReadiness.ps1 hard block Batch、RED/GREEN、commit、localFullLoopGate、threadRolloverGate、nextModuleRunCandidate、L8 remainder。
08

停止后的残留清理机制

当自动化醒来发现“已有线程在工作”而停止时，它不能随手删东西；必须先盘点、分类、再只清理可证明安全的残留。

盘点脚本

Test-ModuleRunV2StoppedAutomationHygiene.ps1 默认只读，检查 automation lease、Codex worktree root、system temp 里的 dry-run handoff 目录。

clean无残留或无需要处理的对象。
cleanup_available发现可安全清理对象，但未执行清理。
cleanup_completed显式 -Cleanup 后完成安全清理。
stop_dirty_worktree有未提交改动，必须人工检查。

清理边界

✓
可清理expired clean lease、stale clean automation worktree、tiku-autopilot-handoff-* temp dir。
×
不可清理active lease、invalid lease、dirty worktree、仓库源码、env 文件、依赖文件、schema、migration、evidence logs。
!
路径约束所有删除都必须在 Codex automation root 或系统 temp root 内完成。
09

模块推进模型：需求到模块到任务到批次

Module Run v2 把高级版推进拆成 execution module，再由每个 Module Run 内部批量推进 3-6 个小 Batch。

execution module 当前定位 目标 localFullLoopGate 关键阻断
authorization-and-access 用户、组织、授权上下文基础模块；Batch 94-100 是历史基线。 L4 local API or Server Action contract 不得改变真实 permission model，不新增角色/权限/配额。
ai-task-and-provider 下一候选模块；先做 provider-agnostic lifecycle，不执行真实 provider。 L2 unit 起步，后续桥接本地体验链 provider/env/secret、raw prompt/response、Cost Calibration Gate blocked。
personal-learning-ai 学生侧个人学习 AI 体验链。 L5 local UI/browser 不生成、不保存、不暴露原始 AI 内容。
organization-training 组织训练、管理员与员工角色流。 L6 role_flow 不得暴露员工主观作答文本或完整 paper 内容。
organization-analytics 聚合统计和隐私保护分析。 L5 local UI/browser 不得暴露非聚合员工细节或私有作答 payload。
ops-governance-and-retention 运营治理、保留、恢复边界、redacted reference。 L6 role_flow 不得执行真实删除、恢复、清理或破坏性数据动作。
10

硬阻断边界

这些边界是自动驾驶稳定性的核心：宁可停下，也不能悄悄越界。

blocked

Cost Calibration Gate

仍保持 blocked，不执行 provider 成本测量或真实样本调用。

blocked

env / secret

不读、不写、不复制、不记录 `.env.local`、key、token、数据库 URL。

blocked

schema / migration

不改 `src/db/schema/**`、`drizzle/**`，不执行 destructive data action。

blocked

provider / deploy

不触碰 real provider、staging/prod/cloud/deploy/payment/external-service。

11

技能、工具与落地手段

机制落地不依赖单一工具，而是由 Codex app capability、仓库脚本、Git Hooks 和技能调度共同完成。

手段 用于什么 落地位置
Codex app automation 按配置定时唤醒，在新线程/工作环境中启动自动化运行。 Codex 应用层；项目内通过 project-state.yaml 记录 id、schedule、status、executionEnvironment。
Thread tools 当 launch policy 允许时，由 agent layer 创建或发送新线程。 create_thread / send_message_to_thread 等 Codex 线程工具。
Skill dispatch 根据任务选择浏览器、Playwright、文档、设计、安全等能力。 docs/04-agent-system/sop/skill-dispatch-matrix.md 与 Codex skills。
PowerShell gates 把流程判断机器化，返回 pass/fail 和决策标签。 scripts/agent-system/*.ps1。
Husky hooks 在 commit/push 时强制执行本地硬阻断。 .husky/pre-commit、.husky/pre-push、post-commit advisory。
Evidence files 把“为什么能继续/为什么必须停”沉淀为可审查事实。 docs/05-execution-logs/task-plans、evidence、audits-reviews。
12

操作手册：如何判断自动驾驶是否能继续

给人和 Agent 的共同检查清单。

自动化醒来时

1
先跑 startupTest-ModuleRunV2AutomationStartupReadiness.ps1。
2
确认 leaseno_active_lease 或 expired_lease_reclaimable 才能继续；active lease 必须停。
3
确认任务只接 pending 且依赖完成的任务，或继续 in_progress 的当前任务。
4
确认 scopeplanned files 必须完全匹配 allowedFiles，不得碰 blockedFiles。
5
确认 thread需要新线程时先生成 handoff，再由 launch policy 决定是否调用线程工具。

任务收尾时

1
跑 focused tests机制任务跑 smoke；业务任务先 RED 后 GREEN。
2
跑全局门禁lint、typecheck、diff、prettier、anchor check。
3
写 closeout evidenceBatch、RED/GREEN、commit、localFullLoopGate、threadRolloverGate、nextModuleRunCandidate、L8 remainder。
4
跑 closeoutTest-ModuleRunV2ModuleCloseoutReadiness.ps1。
5
push 前确认Test-ModuleRunV2PrePushReadiness.ps1，Husky pre-push 还会再跑一次。

当前下一步

当前机制收口后，startup readiness 的合理结果应是 prepare_next_task，下一队列候选仍是 module-run-v2-ai-task-and-provider-planning。它可以被准备为下一轮 Module Run v2 规划，但不能自动越过 provider/env/secret、staging/prod/cloud/deploy、payment、external-service 或 Cost Calibration Gate。
