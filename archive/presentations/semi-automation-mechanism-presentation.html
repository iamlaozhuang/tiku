Tiku Agent Operating Model

把 AI 开发从“会聊天”变成“可恢复、可审查、可交付”的工程流程

这套半自动化推进机制不是让 Agent 自由发挥，而是给它装上任务队列、分支隔离、风险门禁、本地验证、证据记录和人工决策点。简单说：让 AI 帮团队持续推进，但每一步都能被人看懂、复查、接手和回滚。

1 一个任务默认对应一个可审查提交

01

一页讲明白：这套机制到底是什么

它是一个“队列驱动 + Git 隔离 + 本地门禁 + evidence 审计 + 人工高风险审批”的协作系统。

半自动化不是“少管一点”，而是“把该管的地方固化成流程”。

AI 负责推进

读取规范、领取任务、开分支、实现、验证、写 evidence、准备交付。

人负责决策

定义方向、批准高风险动作、判断产品体验、决定是否 push、deploy 或连接真实环境。

02

为什么要这样设计

项目越往后，真正的风险不是 Agent 不会写代码，而是它把上下文、边界、验证和责任混在一起。

防上下文漂移

不能靠聊天记忆继续开发。会话恢复必须读 `project-state.yaml`、`task-queue.yaml`、最新 evidence 和 task plan。

防范围扩散

每个任务声明 `allowedFiles` / `blockedFiles`，避免把依赖、格式化噪声、业务代码、部署动作混进同一个提交。

防不可审查

所有关键结论都要有命令输出、文件路径、分支状态和 evidence。团队成员不需要猜 Agent 做过什么。

03

设计思路：五条底层原则

这些原则决定了机制为什么看起来“有点严格”，也解释了它为什么能跨会话、跨任务继续推进。

1. 状态外置 关键上下文写进仓库文件，而不是留在对话里。
2. 小步可审 一个任务、一个边界、一个 evidence，必要时一个提交。
3. 风险前置 依赖、env、云、prod、provider、破坏性操作默认先 blocked。
4. 证据闭环 先运行门禁和记录输出，再得出结论。
5. 人机分权 Agent 推进确定性工作，人保留高风险决策权。
04

实现方式：用仓库文件搭出一个“任务操作系统”

它没有引入新平台，核心就是一组约定清晰的 Markdown、YAML、PowerShell 脚本和 Git 操作。

控制面

文件 作用
AGENTS.md 全局命名、Git、AI 执行纪律和半自动化硬流程。
automation-loop.md Agent 每次启动、领取、执行、验证、提交、closeout 的 SOP。
project-state.yaml 当前阶段、当前任务、仓库 SHA、质量门禁和 handoff。
task-queue.yaml 任务队列、依赖、允许文件、阻断文件、验证命令和状态。

证据面

文件 作用

local-ci.md 本地质量门禁标准：lint、typecheck、test、format、readiness。
05

任务生命周期：从 pending 到 closed

每个任务都像一张有边界的工单。Agent 不能跳过状态，也不能在 master 上直接开发。

恢复上下文 读规范、ADR、project-state、queue、latest evidence。
领取任务 只处理 pending 且依赖完成的任务。
开短分支 使用 codex/、feat/ 或 fix/ 前缀。
写方案 task plan 记录范围、读源、方案和风险。
执行验证 按 validationCommands 和本地门禁跑。
记录证据 evidence 写命令、结果、禁区和残余缺口。
提交收口 commit、merge、push、cleanup 分开决策。

06

落地细节：团队成员每天实际看什么

新成员不需要先读完整历史，只要掌握这几个入口，就能判断当前项目是否可继续、下一步做什么。

入口 1

先看状态

project-state.yaml 告诉你当前阶段、任务、上次 evidence 和下一步建议。

入口 2

再看队列

task-queue.yaml 告诉你任务是否 pending、依赖是否满足、哪些文件能动。

入口 3

核对证据

evidence/*.md 是判断“是否真的通过”的唯一依据，不以口头描述为准。

入口 4

检查门禁

blocked-gates.yaml 告诉你哪些动作现在默认不能做，需要人明确批准。

07

风险门禁：什么事 Agent 不能自作主张

这套机制最大的价值之一，是把“以后可能出事”的动作提前变成显式审批点。

门禁 默认状态 为什么危险 解除条件
real provider / staging / prod blocked 可能产生费用、泄漏数据、污染环境或误触生产。 明确目标环境、provider、额度、脱敏、日志、回滚和人工批准。
dependency change blocked_by_default 依赖变更会影响供应链、构建、许可证、锁文件和运行面。 按 dependency gate 写清包名、版本、用途、替代方案、风险和批准证据。
secret / env blocked_by_default 环境文件和密钥一旦被读出或写入 evidence，就会变成长期泄漏风险。 明确环境、secret owner、存储位置、脱敏策略、回滚和人工批准。
deploy / cloud blocked_by_default 部署和云资源动作会改变外部世界，不是本地代码验证能覆盖的。 独立部署计划、资源清单、回滚边界、监控责任和人工批准。
destructive data blocked_by_default 删除、重置、迁移、schema push 可能造成不可逆数据损坏。 精确目标、备份/恢复点、验证命令、回滚责任和人工批准。
08

证据体系：为什么说“证据先于结论”

evidence 不是流水账。它是团队复盘、接手、审查和判断风险的最小事实集合。

每份 evidence 应该先给摘要

✓
Resultpass / fail / blocked / pending validation。
✓
Scoperead_only / docs_only / implementation / local_verification / closeout / blocked_gate。
✓
Changed surfaces列出实际变更面，避免“顺手改了别的”。
✓
Gates记录 lint、typecheck、test、build、e2e、readiness、git inventory 的结果。
✓
Forbidden scope明确 env、依赖、schema、migration、staging/prod/cloud、deploy、real provider 没碰或为什么 blocked。

常用本地门禁

Test-AgentSystemReadiness.ps1 检查机制文件、脚本、技能路径和基础能力是否存在。
Test-GitCompletionReadiness.ps1 输出分支、改动、ahead/behind、untracked 和 compare 文件清单。
npm run lint 代码风格和静态规则。
npm run typecheck TypeScript 类型完整性。
npm run test:unit / test:e2e 按任务声明覆盖单元或端到端验证。
git diff --check 空白、行尾、补丁基本健康检查。
09

人机协作：适配当前团队习惯的工作方式

这套机制承认一个现实：人希望 Agent 主动推进，但不能越过关键边界。

你可以这样说

“继续队列”

Agent 应先恢复状态、报告分支/队列/blocked gates/latest evidence，再按 pending 任务推进。

你可以这样说

“先报告再等我”

Agent 只能输出 startup report，不得开分支、改文件、启动服务、push 或接触外部服务。

你可以这样说

“本地真人体验验证”

Agent 只能使用 localhost / 127.0.0.1，按角色和路由矩阵陪跑，不能触碰 staging/prod/provider/env。

Phase 15 的关键升级

增加了启动报告协议、closeout 对账、本地真人体验验证 SOP、blocked gates 注册表、evidence 摘要模板和 `taskKind` 轻量标签。它们共同解决的是：会话恢复看不清、任务关闭后状态滞后、高风险门禁散落、证据太长难读、任务类型不够直观这些问题。

10

团队落地手册：成员按这张清单执行

这部分可以直接作为团队协作约定使用。

开始一个任务前

1
确认 Git 状态当前分支、master 是否 clean、是否与 origin/master 对齐。
2
确认任务来源任务是否在 queue，依赖是否完成，是否有 blocked gate。
3
确认可改文件只动 allowedFiles，不碰 blockedFiles。
4
确认风险审批依赖、env、云、provider、deploy、破坏性动作必须先有人批。
5
确认执行方案task plan 写清实现思路和风险防御后再改实际内容。

结束一个任务前

1
跑任务门禁按 validationCommands 和 local-ci 执行，未跑必须说明原因。
2
写 evidence摘要、命令结果、禁区自检、残余缺口都要落文件。
3
做 Git inventory确认实际变更文件和任务范围一致。
4
分开决策commit、merge、push、PR、deploy、cleanup 是不同动作，不互相默认授权。
5
更新状态closeout 后 project-state 不应保留“待 push/待清理”等过期 handoff。

给团队的一句话版本

Tiku 的半自动化推进机制，本质是把 Agent 的执行力装进工程化轨道：任务从队列来，改动在分支里，风险先拦住，结论看证据，推送和部署由人决定。
