# 文档生命周期与目录结构使用规范 (Doc Management)

> **重要指令**：任何接手此项目的 AI Agent，在读取本文件后，必须严格按照以下目录职责进行文件查找、创建和归档操作。绝对禁止在错误的位置生成或修改文件。

## 1. 全局目录架构职责地图 (Directory Map)

整个项目遵循 **Agentic SDLC（智能体驱动的软件开发生命周期）**，以流水线的形态管理文件：

- **`01-requirements/` (需求工程区)**
  - 职责：存放业务边界和需求拆解。
  - `modules/`：宏观的一级业务模块定义（如 `01-user-auth.md`）。
  - `stories/`：微观的可执行用户故事（如 `epic-01-user-auth.md`），Agent 接单时直接参考此处的 AC（验收标准）。

- **`02-architecture/` (架构工程区)**
  - 职责：存放技术边界、防劣化控制和系统设计。
  - `system-design/`：UI 线框图、设计 Tokens 等。
  - `interfaces/`：API 契约和数据结构定义。
  - `adr/`：架构决策记录（Architecture Decision Records）。Agent 在推翻现有设计前必须阅读此处的决策历史。

- **`03-standards/` (规范工程区)**
  - 职责：全局质量底线（即当前您正在阅读的目录）。
  - 必须严格遵守此目录下的 `testing-tdd.md`（红绿重构规则）、`coding-style.md`（反 AI 味规则）等。

- **`04-agent-system/` (智能体编排区)**
  - 职责：定义多 Agent 协作的 SOP（标准操作规程）、提示词模板和项目阶段性里程碑。

- **`05-execution-logs/` (任务执行沙盒)**
  - 职责：记录正向开发价值流的痕迹。
  - `task-plans/`：**关键！** Agent 在任何编码前，必须在此处创建如 `2026-05-14-xxx-plan.md` 的执行方案。

- **`06-issue-tracking/` (缺陷追踪区)**
  - 职责：记录逆向价值流（Bug、测试失败修复记录、暂缓的技术负债）。

- **`archive/` (历史档案馆)**
  - 职责：被废弃的规范、过时的计划和演示文件。Agent 严禁以 `archive/` 中的文件作为当前开发的依据（防止幻觉）。

## 2. 文档生命周期操作规范

1. **命名铁律**：所有新文档文件名必须采用 `kebab-case.md`。
2. **状态前缀**：
   - 作为永续治理规范的文件（如本文件、`coding-style.md`）**禁止**加日期前缀。
   - 具有时效性或阶段性的操作方案（如 `task-plans` 下的文件），**必须**加 `YYYY-MM-DD-` 前缀。
3. **归档原则**：当某个需求或规范被重构作废时，Agent 不能直接删除，必须将其移动至 `archive/` 目录下相应的子分类中，并附带一条废弃说明。
