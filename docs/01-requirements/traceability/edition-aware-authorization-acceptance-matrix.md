# Edition-Aware Authorization Acceptance Matrix

## Purpose

Record acceptance scenarios for personal and organization `standard | advanced` authorization. This matrix is a requirements and validation planning artifact only; it does not approve schema, migration, API, source code, env/secret, provider, payment, deployment, or Cost Calibration Gate work.

## Scenario Matrix

| scenarioId                         | Actor                 | Setup                                                                                                                                   | Expected requirement outcome                                                                                                                     | Blocked implementation gates                                 |
| ---------------------------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------ |
| `EAA-PERSONAL-STANDARD-ACTIVATION` | `student`, `admin`    | Platform operations issues a `personal_standard_activation` `redeem_code`; the student redeems it for matching `profession + level`.    | A standard `personal_auth` is available; `edition = standard`; `effectiveEdition = standard`; advanced-only capabilities stay blocked.           | Schema, API, service, UI, e2e.                               |
| `EAA-PERSONAL-ADVANCED-ACTIVATION` | `student`, `admin`    | Platform operations issues a `personal_advanced_activation` `redeem_code`; the student redeems it for matching `profession + level`.    | An advanced `personal_auth` is available; `edition = advanced`; `effectiveEdition = advanced`; quota ownership is personal.                      | Schema, API, service, UI, quota, provider, Cost Calibration. |
| `EAA-PERSONAL-EDITION-UPGRADE`     | `student`, `admin`    | Student has an active standard `personal_auth`; platform operations issues an `edition_upgrade` code for matching `profession + level`. | Redemption creates `auth_upgrade`; no new `personal_auth` is created; `effectiveEdition = advanced` while source and upgrade remain valid.       | Schema, API, service, UI, audit.                             |
| `EAA-PERSONAL-UPGRADE-MISMATCH`    | `student`             | Student attempts `edition_upgrade` without matching active standard `personal_auth`.                                                    | Upgrade fails with a safe validation result; no `auth_upgrade` is created; no advanced capability is enabled.                                    | Schema, API, service, UI.                                    |
| `EAA-PERSONAL-DUPLICATE-UPGRADE`   | `student`             | Student already has effective advanced access for the same `profession + level`.                                                        | A second upgrade is rejected or left idempotent without consuming another upgrade code or creating duplicate active upgrade state.               | Schema, API, service, audit.                                 |
| `EAA-ORG-STANDARD-CREATION`        | `admin`               | Platform operations creates `org_auth` with `edition = standard`.                                                                       | Employees in scope can use standard organization authorization only; advanced organization capabilities stay blocked.                            | Schema, API, service, UI, e2e.                               |
| `EAA-ORG-ADVANCED-CREATION`        | `admin`               | Platform operations creates `org_auth` with `edition = advanced`.                                                                       | Employees in scope receive organization advanced context while the `org_auth` remains valid; quota ownership is organization.                    | Schema, API, service, UI, quota, e2e.                        |
| `EAA-ORG-MANUAL-UPGRADE`           | `admin`               | Active standard `org_auth` is upgraded by platform operations with external reference and operations note.                              | `auth_upgrade.source_type = ops_manual` records the upgrade; `effectiveEdition = advanced`; original `org_auth` is unchanged.                    | Schema, API, service, UI, audit.                             |
| `EAA-UPGRADE-EXPIRED-FALLBACK`     | `student`, `employee` | An upgrade expires while the source authorization remains valid.                                                                        | `effectiveEdition` falls back to `standard` unless another valid advanced authorization applies.                                                 | Schema, API, service, e2e.                                   |
| `EAA-UPGRADE-REVOKED-FALLBACK`     | `admin`               | Platform operations revokes an active upgrade.                                                                                          | Revocation is auditable; source authorization remains unchanged; future advanced access is blocked unless another valid advanced source applies. | Schema, API, service, UI, audit.                             |
| `EAA-CONTEXT-NO-AUTO-SWITCH`       | `student`, `employee` | User has multiple contexts with different edition or quota state.                                                                       | The system does not automatically switch context to obtain higher edition or more quota; selection remains explicit or workflow-owned.           | API, service, UI, e2e.                                       |
| `EAA-QUOTA-INSUFFICIENT`           | `student`, `employee` | Context is advanced but quota owner lacks available quota.                                                                              | Advanced action is blocked by quota without changing authorization edition; safe summary identifies quota owner type only.                       | Quota, service, UI, Cost Calibration.                        |
| `EAA-REDACTED-AUDIT`               | `admin`, `auditor`    | Authorization or upgrade operation is reviewed in operations evidence.                                                                  | Evidence includes public ids, status, timestamps, edition, `effectiveEdition`, quota owner type, and redacted audit references only.             | Audit, mapper, UI.                                           |

## Acceptance Boundaries

- Every scenario must preserve the service-layer authorization boundary from ADR-002.
- No scenario authorizes online payment, external purchase confirmation, provider calls, env/secret work, schema migration, deployment, or production quota defaults.
- Evidence must never contain secret material, provider payloads, raw prompts, raw generated AI content, raw employee answer text, full paper content, internal database rows, or plaintext `redeem_code` values.
