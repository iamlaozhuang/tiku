# Edition-Aware Authorization Acceptance Matrix

## Purpose

Record acceptance scenarios for personal and organization `standard | advanced` authorization. This matrix is a requirements and validation planning artifact only; it does not approve schema, migration, API, source code, env/secret, provider, payment, deployment, or Cost Calibration Gate work.

## Scenario Matrix

| `EAA-UPGRADE-EXPIRED-FALLBACK`       | `student`, `employee` | An upgrade expires while the source authorization remains valid.                                                                                              | `effectiveEdition` falls back to `standard` unless another valid advanced authorization applies.                                                 | Schema, API, service, e2e.                                   |
| `EAA-UPGRADE-REVOKED-FALLBACK`       | `admin`               | Platform operations revokes an active upgrade.                                                                                                                | Revocation is auditable; source authorization remains unchanged; future advanced access is blocked unless another valid advanced source applies. | Schema, API, service, UI, audit.                             |
| `EAA-SOURCE-EXPIRED-BLOCKED`         | `student`, `employee` | Source `personal_auth` or `org_auth` is outside its valid date range.                                                                                         | No valid authorization context is produced; `effectiveEdition` is not elevated by stale source or upgrade state.                                 | Schema, API, service, UI, e2e.                               |

| `EAA-CONTEXT-NO-AUTO-SWITCH`         | `student`, `employee` | User has multiple contexts with different edition or quota state.                                                                                             | The system does not automatically switch context to obtain higher edition or more quota; selection remains explicit or workflow-owned.           | API, service, UI, e2e.                                       |

| `EAA-REDACTED-AUDIT`                 | `admin`, `auditor`    | Authorization or upgrade operation is reviewed in operations evidence.                                                                                        | Evidence includes public ids, status, timestamps, edition, `effectiveEdition`, quota owner type, and redacted audit references only.             | Audit, mapper, UI.                                           |

## Acceptance Boundaries

- Every scenario must preserve the service-layer authorization boundary from ADR-002.
- No scenario authorizes online payment, external purchase confirmation, provider calls, env/secret work, schema migration, deployment, or production quota defaults.
- Evidence must never contain secret material, provider payloads, raw prompts, raw generated AI content, raw employee answer text, full paper content, internal database rows, or plaintext `redeem_code` values.
- The eligible operations UI plaintext visibility decision for `redeem_code` distribution, list, and detail surfaces does not relax the evidence, log, screenshot, export, or committed-document redaction boundary.
