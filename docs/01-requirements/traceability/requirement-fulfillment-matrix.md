## Status Rules

- `implemented` in historical matrices maps to `covered` only when a traceable validation record exists.
- Code, mock, fixture, or documentation without a verified user path remains `partial`, `gap`, `blocked`, or `not_applicable`.
- `freshEvidenceRequired=yes` means a later approved `local_full_flow` or targeted verification task must still prove the user path.
- Supplemental audit rows use `AUDIT-0621-*` and carry `source_freshness_conflict` when they came from current user questions or newly observed source conflicts instead of a formal requirement ID.

## Static Findings From Current Questions

- `theory` and `skill` are canonical `subject` values, not two different navigation modes.
- The home page groups papers by `subject`; the displayed set count is the number of `paper` records in that group. The student paper request uses `pageSize=20`, which is page size, not the number of questions in one paper.
- Current schema and composition code allow many `paper_question` rows per `paper`; no product-level maximum was found in the static scan.
- Canonical question types are `single_choice`, `multi_choice`, `true_false`, `fill_blank`, `short_answer`, `case_analysis`, and `calculation`.
- Compatibility for `multiple_choice` to `multi_choice` and `subjective` to `short_answer` still exists in student practice/mock runtime paths; this conflicts with the expectation that all names are fully normalized and needs an explicit decision.
- Current `org_auth` creation and schema are single `profession`, single `level`, and no `subject` scope. Multiple organizations can be referenced, but that is not the same as multiple professions or levels.
- Creating multiple `org_auth` rows for one enterprise appears to share the same `organization`/enterprise surface, but overlap and occupancy rules need product confirmation and runtime verification.
- No content_admin AI question generation or AI paper generation entry was found. Student personal AI generation exists separately and is advanced-scoped/gated.
