## Recovery Protocol

Future requirement, UI/UX, or implementation work in this topic family must use this recovery order after the project
governance files:

1. Read `docs/01-requirements/00-index.md`.
2. Read `docs/01-requirements/advanced-edition/00-index.md`.
3. Read `docs/01-requirements/advanced-edition/edition-aware-authorization-requirements.md` and ADR-007 for edition or
   authorization work.
4. Read `docs/01-requirements/traceability/2026-07-02-role-auth-training-ops-decision-package.md`.
5. Read this ledger and cite the relevant `CT-REQ-*` row ids in any follow-up task plan, UI/UX contract, implementation
   task, evidence, or audit.

Anti-context-bloat rules:

- Do not rely on conversation memory after compaction; recover from this ledger, the decision package, evidence, and the
  current git diff.
- Every new decision must add or update exactly one ledger row before implementation starts.
- Every changed or superseded older requirement must name its source and the superseding ledger row.
- Implementation work must not mark a row as implemented unless that specific task validates source/runtime behavior.
- UI/UX-sensitive rows must produce a design contract before product source work starts.

## Classification Model

| Class                  | Meaning                                                                                    |
| ---------------------- | ------------------------------------------------------------------------------------------ |
| `existing_no_change`   | Requirement already existed and the current discussion did not change its semantics.       |
| `clarified`            | Existing requirement was clarified without changing the product direction.                 |
| `supplemented`         | Current discussion added detail that was missing or too implicit.                          |
| `changed`              | Current discussion superseded older wording or implementation assumptions.                 |
| `new_decision`         | Current discussion created a decision not previously recorded in requirement sources.      |
| `implementation_gap`   | Static source inspection or requirement comparison found source/runtime work still needed. |
| `uiux_contract_needed` | Product direction is decided, but interaction/design contract must be produced first.      |
| `process_guard`        | A process or recovery decision that governs later work.                                    |

Implementation status values:

- `docs_only`: this row is a requirement/process record only.
- `not_checked_current_task`: implementation was not inspected for this row in the current task.
- `implemented_static_seen`: static inspection found source that appears to represent the behavior, but runtime was not
  revalidated by this task.
- `partial_static_seen`: static inspection found partial source support and remaining gaps.
- `not_implemented_static_seen`: static inspection found the confirmed behavior missing or contradicted.

## No-Omission Controls

- Every current-thread confirmed requirement is represented by at least one `CT-REQ-*` row, including supplemental rows
  `CT-REQ-031` through `CT-REQ-043`.
- Existing decisions that were not changed are marked `existing_no_change` or `clarified`, preventing duplicate work.

  requirements.
- Source/runtime reality is separated from requirement decisions through the implementation status column.
- UI/UX-sensitive items are explicitly marked `uiux_contract_needed`; they should not move directly into product source
  changes.
- Future task plans must cite row ids, so context compaction cannot drop the decision trail.
