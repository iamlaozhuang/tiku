# Unified Edition Delta Matrix

## Scope

This matrix is derived from:

- `docs/01-requirements/traceability/unified-standard-advanced-source-index.md`
- `docs/01-requirements/traceability/capability-catalog.md`
- `docs/01-requirements/use-cases/use-case-catalog.md`

It compares standard edition MVP and advanced edition requirement scope only. It does not create a technical landing
matrix, code audit, implementation queue, schema, migration, provider work, env/secret work, e2e work, deployment, PR,
force-push, payment, external-service work, or Cost Calibration Gate execution.

## Field Model

| Field                     | Meaning                                                                                                              |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| `deltaId`                 | Stable audit id for one standard-vs-advanced delta.                                                                  |
| `capabilityIds`           | Capability catalog ids compared by this row.                                                                         |
| `useCaseIds`              | Use case catalog ids compared by this row.                                                                           |
| `sourceIds`               | Frozen source index ids that justify the comparison.                                                                 |
| `standardMvpPosition`     | Requirement position for the standard MVP.                                                                           |
| `advancedEditionPosition` | Requirement position for the advanced edition.                                                                       |
| `deltaStatus`             | `shared_core`, `advanced_extension`, `standard_non_goal`, `future_non_goal`, `audit_only`, or `blocked_gate`.        |
| `blockedGates`            | Gates that still block implementation or execution.                                                                  |
| `conflictRefs`            | `CFX-*` ids from the source index that must be carried forward.                                                      |
| `auditUseOnly`            | `true` means the row is only a governance/audit reference and cannot seed implementation from this matrix.           |
| `implementationEligible`  | Whether a future scoped task may consider implementation after all declared gates and approvals are satisfied.       |
| `notes`                   | Boundary summary; no raw secret, provider payload, raw response, database URL, row data, prompt, or private content. |

`implementationEligible` records future eligibility only. It is not implementation, provider, env/secret, staging,
deployment, payment, schema, migration, e2e, PR, force-push, or Cost Calibration approval.

## Edition-Aware Authorization Supplement

The edition-aware authorization decision refines `DELTA-AUTH-ACCOUNT-SESSION`, `DELTA-PERSONAL-AUTH`,
`DELTA-ORG-AUTH-PORTAL`, and `DELTA-OPS-QUOTA`:

- Standard remains the compatibility baseline for existing unversioned authorization records.
- Advanced can be issued directly on new personal or organization authorization sources.
- Standard-to-advanced upgrades are represented by `auth_upgrade`; the source authorization `edition` is not overwritten.
- Personal `edition_upgrade` is a `redeem_code` kind, while organization upgrade is a platform operations action.
- Quota owner follows the selected authorization context; exact production quota values remain blocked by Cost Calibration.
- Payment, pricing, invoice, refund, external-service confirmation, provider execution, and deployment remain non-goals.

## Delta Use Rules

1. Later technical matrix or code audit tasks must cite `deltaId` together with `useCaseId`, `capabilityId`, and
   `sourceId` values.
2. `auditUseOnly: true` and `implementationEligible: false` rows must not seed implementation tasks.
3. `blocked_gate` rows describe future approval packages only; they do not approve provider, env/secret,
   staging/prod/cloud/deploy, payment, external-service, schema/migration, package/lockfile, e2e, PR, force-push, code
   audit, code fix, or Cost Calibration work.
4. `future_non_goal` rows must not be converted into standard MVP or advanced first-release scope.
5. `CFX-*` conflict references must remain unresolved unless a later adjudication task explicitly authorizes a
   decision.
