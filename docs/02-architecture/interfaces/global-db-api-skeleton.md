## ID Strategy

- Internal database primary key: `id`.
- External route identifiers use `public_id` in database rows and `publicId` in API JSON.
- External URLs do not expose internal `id`.
- `public_id` is only for resources that must be visible in external URLs. It is not required by default for every table.
- public_id is not an authorization mechanism, security boundary, or anti-horizontal-access-control mechanism.
- Every read or mutation by `public_id` must still pass session, role, authorization scope, and resource ownership checks.
- `public_id` must be unique within the resource type that exposes it.
- `public_id` must be non-sequential and non-guessable enough for URL use.
- `public_id` is generated by the server.

## API Contract

- Route Handlers and Server Actions are thin adapters only. They validate transport input, call services, map service results to API response contracts, and return the standard response.
- Route Handlers and Server Actions must not return database rows directly.
- API response contract definitions belong in `src/server/contracts`.
- Database row to API JSON conversion belongs in `src/server/mappers`.
- Database fields remain `snake_case`; API JSON fields are converted to `camelCase` by server-side mappers.
- Pagination responses use `total` as the standard total-count field.

## Cross-Module Rules

- Published papers are immutable.
- Published paper questions use snapshots.
- Mock exam reports store snapshots.
- Student-side content access requires a current valid authorization.
- Student-side historical report access requires a current valid authorization.
- Effective access is the union of `personal_auth` and `org_auth`.
- Organization hierarchy and purchase or authorization relationships are modeled separately.
- When an employee changes organization, historical learning records keep the organization ownership snapshot from answer time.
- After an employee leaves and is unbound from an organization, reports produced only by that organization's authorization are no longer visible unless the user obtains the corresponding authorization again.
- First-phase `org_auth` ranges for the same `profession` and `level` must not overlap.
- Published papers cannot be deleted.
- Papers with existing `answer_record` rows cannot be deleted.
- Source questions are locked after being referenced by any published paper.
- Repeated AI scoring is idempotent.
- RAG citations are not fabricated.
- Critical admin operations write audit logs.
