## Local-Only And Blocked-Gate Policy

- Local mock and deterministic provider validation may continue.
- `real-provider-staging-redaction` remains blocked.
- `dependency-change` remains blocked by default until a task-specific approval record exists.
- `secret-env-change` remains blocked by default; do not read `.env.local` or modify `.env.example`.
- `deploy-and-cloud-change` remains blocked by default.
- `destructive-data-operation` remains blocked by default.
- Evidence may record that `.env.local` exists only when framework logs expose that fact; it must not read or copy values.

## Recommended Execution Order And Stop Points

1. Close Phase 21 planning and stop for human confirmation.
2. Start low-risk UI/evidence tasks that do not require approvals.
3. Stop before the first `auth_permission_model` task and obtain approval.
4. Stop before the first `database_migration` task and obtain approval.
5. Stop before dependency-driven import/conversion/RAG tasks and obtain dependency approval.
6. Keep provider/env/cloud/deploy gates blocked unless the user explicitly opens a separate task with required evidence.
7. After each subphase, run re-audit/evidence-only reconciliation before claiming the next high-risk wave.
