## Three-Round Review

### Round 1 - Requirement Fit

- RAG 需求要求资源按专业/等级过滤，并支持知识点体系治理。
- 2026-07-06 AI 组卷合同要求知识覆盖不能只有自由文本，必须结构化选择优先。
- 当前 G1-G4 均直接阻断该合同闭环。

### Round 2 - Implementation Risk

- 最大风险是把知识点 public id 当作权限边界。修复必须只把它作为请求范围，权限仍由现有授权和服务检查决定。
- 第二风险是多知识点正式题查询只取第一个节点。AI 组卷分支必须实现多节点查询去重或等价收窄，避免漏题。
- 第三风险是让标准版看到高级入口。回归分支必须复核标准版拒绝/升级提示。

### Round 3 - Test Adequacy

- G1 需要 UI 提交 FormData 和 DTO 映射测试。
- G2/G3 需要页面参数面板测试，证明指定知识点模式可选、无选项时禁用。
- G4 需要服务测试，证明 AI 组卷 RAG grounding 与正式题库查询消费 `knowledgeNodePublicIds`。
- 收口分支需要全角色矩阵复核，覆盖空态、错误态、禁用态、权限和 edition 边界。

## Decision

可以进入源码修复。必须先写 failing tests，再做最小实现；每个分支完成后写独立 evidence/audit、合入 `master`、跑门禁、推送和清理短分支。
