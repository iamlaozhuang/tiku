## 1. 审批结论

### 1.1 P0 数据完整性

建议批准为下一轮实现基线，且必须先于一般体验优化实施。

当前基线存在一条可由源码与现有测试闭合的风险链：新建表单把演示文案作为真实表单值，客户端保存门禁没有区分“用户已完成录入”和“系统预填演示内容”，服务端只要收到非空且类型合法的载荷就会接受，现有单元测试还把这些值作为成功创建样本。即使本任务没有进行浏览器或数据库写入，这已经足以证明“误保存占位内容”的当前合同风险，而不是历史推测。

该结论不依赖禁止某几个固定中文文案。正确修复方向是让新建表单从语义空值开始，并由客户端与服务端共同验证真实业务完整性。

### 1.2 平台级交互契约

建议批准为后续优化的规范基线，但应先固化规范等级和例外机制，再分批推广。

现有稳定需求已经规定后台分页、排序、筛选自动刷新、URL 状态恢复、确认、Toast 和并发冲突；2026-07-07 全角色 UI/UX 基线又补充了工作区上下文、状态模板、列表/详情分离、内容生命周期和任务容器方向。因此，本包不是重新发明平台体验，而是把已存在的分散规则归一为可测试合同。

平台一致性不等于所有页面使用同一种容器。应统一的是状态语义、按钮层级、验证、反馈、焦点、URL 恢复和异步行为；容器必须按任务复杂度选择，并在例外登记中说明理由。

### 1.3 审批边界

产品负责人已批准把契约提升到稳定 requirements/traceability，并批准启动独立 Batch A。该批准允许本 docs-only 任务本地提交、ff-only 合入和清理，也允许 Batch A 的源码/测试实现；不批准浏览器、数据库、Provider、依赖、schema、fixture、staging、production、deploy、PR、force push 或普通 push。

## 2. 第一性原理评估方法

判断一个问题或契约能否作为实现基线，必须同时满足五个条件：

| 维度       | 判断问题                                                       | 本包结论                                 |
| ---------- | -------------------------------------------------------------- | ---------------------------------------- |
| 真实性     | 风险是否来自当前 SHA 的源码/测试，而非旧截图或主观偏好？       | 通过；有当前源码和当前成功测试路径。     |
| 可执行性   | 是否能转化为明确的客户端、服务端和交互验收条件？               | 通过；见 P0 与平台契约矩阵。             |
| 可验证性   | 是否能用失败测试、定向测试、全量门禁和受控验收证明？           | 通过；验证方法已拆分，运行时验收另审批。 |
| 边界安全   | 是否保持角色、edition、authorization、AI、企业训练和脱敏规则？ | 通过；合同只统一体验，不扩大业务能力。   |
| 可逆与可拆 | 是否能独立提交、独立回滚，避免一次性全平台改造？               | 通过；建议拆为六个批次，P0 独立先行。    |

只要任一维度不满足，就不能把建议直接升级为实现基线。

## 4. P0 数据完整性验收矩阵

后续 P0 实现必须全部满足以下硬门禁：

| ID    | 层级       | 验收条件                                                                                                                                                                                    |
| ----- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P0-01 | 新建默认值 | 题干、解析、标准答案、材料标题、材料正文、选项内容和评分点描述不以演示文案作为提交值；示例仅可用 placeholder/helper。无显式创建上下文时，专业、等级、科目等关键分类不得伪装成已由用户确认。 |
| P0-02 | 语义空值   | 纯空白、仅空标签、`<p><br></p>`、不可见字符等富文本空值按空处理；有效文字或具备有效受管引用与可访问说明的媒体/表格节点可计为内容；不得用固定文案黑名单代替语义判断。                        |
| P0-03 | 客户端     | 保存前显示字段级错误和表单摘要，阻止提交，并把焦点移到第一个无效字段；修正后错误可恢复。                                                                                                    |
| P0-04 | 服务端     | UI 被绕过时仍拒绝同一类空值和题型不完整载荷，保持标准 API envelope，不返回堆栈或内部细节。                                                                                                  |
| P0-05 | 单选题     | 至少两个非空、label 唯一的选项，且恰好一个正确项；标准答案与正确项一致。                                                                                                                    |
| P0-06 | 多选题     | 至少两个非空、label 唯一的选项，且至少两个正确项；标准答案与正确项集合一致。                                                                                                                |
| P0-07 | 判断题     | 固定“正确/错误”语义和 A/B 映射，只允许一个正确项；不让演示选项内容进入自由编辑脏状态。                                                                                                      |
| P0-08 | 填空题     | 自动匹配时至少一个有效答案；AI 评分时需要有效评分点；切换方式保留数据但不能绕过当前方式的保存门禁。                                                                                         |
| P0-09 | 主观题     | 不发送选择题选项；评分点描述非空、分值为正且符合 0.5 粒度；参考答案、解析按稳定需求校验。                                                                                                   |
| P0-10 | 材料       | 标题与正文语义非空，正文不超过 30000 字符；空表格模板、破损图片引用或无有效内容的 helper 不能让空正文误判为有效内容。                                                                       |
| P0-11 | 双击与并发 | 提交中禁止重复保存；成功、失败和冲突状态可区分，失败不得清空作者已输入内容。                                                                                                                |
| P0-12 | 回归       | 编辑、复制、停用、锁定、引用摘要、知识点/标签/材料绑定和 true/false 映射保持既有行为。                                                                                                      |
| P0-13 | 可访问性   | 必填说明、错误文本、`aria-invalid`/`aria-describedby`、错误聚焦和键盘提交均可验证；禁用按钮旁有可见原因。                                                                                   |
| P0-14 | 测试       | 先加入会失败的客户端/API/题型矩阵测试；定向测试、全量单测、lint、typecheck、format、build 和 diff 全部通过。                                                                                |

P0 完成定义：客户端与服务端必须同时通过；只清空默认文案、只禁用按钮或只在 UI 校验都不算完成。

## 5. 平台级交互契约

### 5.1 `MUST`：后续所有相关后台优化必须遵守

| 合同 ID | 规范                                                                                                                              |
| ------- | --------------------------------------------------------------------------------------------------------------------------------- |
| PIC-01  | 页面先表达工作区、角色/organization context、内容域和当前任务；UI 可见性不能替代服务端 authorization。                            |
| PIC-02  | 列表的筛选、排序、页码和 pageSize 写入 URL；筛选或 pageSize 改变后回到第 1 页，刷新/返回后恢复。                                  |
| PIC-03  | 筛选变化自动刷新；关键词输入使用短延迟 debounce，取消或忽略过期请求，保证“最后一次意图获胜”。                                     |
| PIC-04  | 统一区分 initial loading、refreshing、no data、filtered empty、error、forbidden、edition unavailable、missing context、conflict。 |
| PIC-05  | 每个页面区域只有一个主动作；次要动作降级；危险动作使用 destructive 语义和二次确认；禁用动作旁显示原因。                           |
| PIC-06  | 创建/编辑使用同一业务验证合同；字段错误就近显示，跨字段错误显示在表单摘要，服务端错误映射到可恢复状态。                           |
| PIC-07  | 提交中防重复；成功提供 Toast 和对象级状态更新，失败保留输入；并发冲突提示刷新后重试，不静默覆盖。                                 |
| PIC-08  | 详情 Drawer 必须具备语义标题、初始焦点、焦点循环、Escape、关闭后焦点恢复；长内容不把主列表挤入竞争层级。                          |
| PIC-09  | 核心资源的长表单新建/编辑使用独立 editor route；离开脏表单、返回列表和刷新行为必须可预期。                                        |
| PIC-10  | 键盘顺序、可见焦点、标签/说明/错误关联、状态宣布和最小目标尺寸纳入验收；不以截图替代可访问性验证。                                |
| PIC-11  | 后台保持 desktop-first，但窄视口必须无页面级横向溢出；宽表只在自身容器滚动。                                                      |
| PIC-12  | 术语、状态、按钮层级和容器选择跨题库、材料、试卷、运营、组织训练保持一致，同时保留学员 mobile-first 与后台 desktop-first。        |
| PIC-13  | AI、edition、authorization、organization training、手机号和 `redeem_code` 的既有服务端、数据域和脱敏边界不得因 UI 统一而改变。    |

### 5.2 `SHOULD`：默认采用，偏离需记录理由

- 显示当前有效筛选 chip、结果数量和一键重置；不把筛选藏在不可见状态。
- 从详情/编辑返回时恢复列表滚动位置、筛选和焦点目标。
- 长表单按业务段分组并提供顶部错误摘要；不使用多列密集表单压缩阅读顺序。
- 对锁定、引用、edition、organization scope 等阻断状态提供安全下一步，如复制为新草稿、返回允许范围或联系负责人。
- 列表刷新使用保留上下文的 updating 状态，不在每次筛选时清空整个页面。

### 5.3 任务容器与允许例外

| 容器                  | 适用任务                                              | 不适用                                                   |
| --------------------- | ----------------------------------------------------- | -------------------------------------------------------- |
| Detail Drawer         | 只读详情、引用关系、审计摘要、轻量上下文查看          | 长表单、复杂校验、富文本编辑                             |
| Dialog                | 单一低风险确认、少量字段动作、危险操作二次确认        | 多步骤创建、需要保存草稿的任务                           |
| Task Drawer           | 有限字段、需要保留列表上下文的短任务                  | 题目/材料完整编辑、试卷组卷、企业训练四步向导            |
| Dedicated editor page | 题目、材料等核心资源的新建/编辑                       | 纯只读详情                                               |
| Composer/Wizard       | 试卷组卷、企业训练、授权包等多步骤且有预览/复核的任务 | 简单属性开关                                             |

## 6. 跨平台一致性抽样矩阵

平台契约推广不能只看题库/材料。每个共享模式至少在以下代表性流程复核：

| 页面族                 | 必查合同                                               | 业务边界                                               |
| ---------------------- | ------------------------------------------------------ | ------------------------------------------------------ |
| 题库题目/材料库        | 列表状态、独立编辑页、语义验证、锁定/复制、详情 Drawer | 正式内容生命周期                                       |
| 试卷与发布             | 列表/详情、Composer、发布 blocker、返回状态            | 发布后锁定与快照                                       |
| 用户/organization/auth | URL 列表、详情、Dialog/Wizard、冲突、危险确认          | authorization、手机号脱敏、角色                        |
| `redeem_code`          | 列表/详情、生成向导、确认、反馈                        | 合资格运营 UI 明文例外；证据/日志继续脱敏              |
| 企业训练               | 列表/详情、四步 Wizard、版本/下架状态                  | `org_advanced_admin`、organization scope、非正式 paper |
| 练习/`mock_exam`/报告  | 状态反馈、继续/重启确认、报告层级                      | 学员 mobile-first；不照搬后台布局                      |
| AI 页面                | 五区结构、结果/历史、blocked/unavailable、异步状态     | Provider/edition/数据域不改变                          |

这里的“抽样”只验证共享合同是否适用，不授权同时改造这些页面。

## 7. 建议实施批次

### Batch A：P0 数据完整性（必须先行）

- 范围：题目/材料新建默认值、共享语义空值、题型完整性、客户端/API 一致验证、回归测试。
- 候选文件：题目材料客户端、question/material validators、`admin-question-material-ui.test.ts`、`phase-9-content-question-material-runtime.test.ts`。
- 方法：TDD；先让现有“默认内容可成功创建”测试转为失败证据，再实现最小闭环。
- 不包含：editor route 重构、全平台组件抽象、DB/schema/fixture、浏览器写入。

### Batch B：平台契约 SSOT 与共享原语

- 范围：把批准后的 PIC-01 至 PIC-13 提升到稳定 traceability；盘点并收敛已有 list query、state template、detail drawer 和 feedback 原语。
- 目标：只建立能被至少两个页面复用的原语，避免先造“大一统表单框架”。
- 停止条件：需要改变业务 authorization 或引入新依赖。

### Batch C：题目/材料独立编辑工作区

- 范围：新建/编辑 route、脏状态离开保护、返回列表恢复、锁定与复制路径；详情继续使用 Drawer。
- 前置：Batch A 完成；route 决策和 wireflow 获得产品批准。
- 回归：列表筛选、引用摘要、停用、复制和现有焦点合同。

### Batch D：列表请求与筛选一致性

- 范围：关键词 debounce、过期请求取消/忽略、updating 状态、有效筛选 chip、URL/焦点/滚动恢复。
- 先在题目/材料证明，再决定是否推广到其他后台列表。

### Batch E：平台推广与例外台账

- 范围：按页面族逐批应用状态、按钮、容器与反馈合同；每批只处理一个工作区或一个共享模式。
- 必须保护：A01-A30、手机号策略、`redeem_code` 明文例外、edition/authorization、AI plan-and-select、企业训练数据域。

### Batch F：全角色验收

- 范围：代码测试通过后，在单独批准的 localhost 任务中验证内容后台代表流程，再复核运营、组织、学员代表流程。
- 浏览器、账号、截图和受控数据创建必须单独批准；凭证只内存使用，证据脱敏，不抓 raw DOM。

## 8. 后续实现统一门禁

1. 使用短分支/worktree，声明精确 allowed files 与 blocked files；

6. commit、ff-only merge、push、浏览器、DB 或 Provider 均按各任务 fresh approval 执行。
