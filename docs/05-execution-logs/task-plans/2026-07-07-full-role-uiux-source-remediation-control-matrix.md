## First-Principles Controls

1. UI visibility is discovery only. Role, `effectiveEdition`, organization context, expiry, revocation, and quota stay service-owned.
2. A source branch may improve structure, wording, state, and layout only inside its matrix scope. Cross-branch findings are recorded in this matrix and must be resolved by branch 8 at the latest.
3. Standard roles must not gain advanced AI or enterprise training capability through menu, route, component, or fallback state.
4. Advanced roles must keep approved entries discoverable unless the branch explicitly replaces them with an equivalent visible path.
5. AI generated output remains in learner, organization training draft, or content review draft domains. It must not directly become formal content or formal learning records.
6. Eligible `ops_admin` and `super_admin` product UI plaintext `redeem_code` behavior is preserved. Evidence, logs, committed docs, screenshots, exports, and non-eligible views remain redacted.
7. No branch may change DB, account material, fixture, Provider execution/configuration, env, dependency, schema/migration/seed, staging/prod/deploy, payment, Cost Calibration, e2e artifacts, screenshots, traces, raw DOM, or private values.
8. Every source branch must start from latest `origin/master`, use a short branch, write a branch-specific task plan, run targeted tests first, run adversarial review, write redacted evidence/audit, commit, fast-forward merge to `master`, run master gates, push, delete the short branch, confirm clean/aligned, then move to the next branch.

## Fixed Per-Branch Flow

1. `git fetch origin`; confirm `master` equals `origin/master` and worktree clean.
2. `git checkout master`; `git pull --ff-only`; create short branch from `origin/master`.
3. Re-read branch-specific baseline, design-board rows, source files, and tests.
4. Create branch task plan in `docs/05-execution-logs/task-plans/`.
5. Write failing targeted test or source-contract assertion first for behavioral/UI structure changes.
6. Implement minimal UI/source changes inside allowed range.
7. Run targeted tests, `tsc --noEmit`, targeted ESLint, scoped Prettier check, full `vitest run`, `git diff --check`, and Module Run v2 precommit.
8. Write redacted evidence and adversarial audit with `Requirement Mapping Result`.
9. Verify no forbidden files changed.
10. Commit one reviewable branch commit.
11. Fast-forward merge to `master`.
12. On `master`, run pre-push readiness and the branch validation gate required by evidence.
13. Push `master` to `origin/master`.
14. Delete the short branch.
15. Confirm worktree clean and `master` equals `origin/master`.
16. Update the control matrix status before starting the next branch.

## Adversarial Risk Checklist

- Standard personal, standard employee, and standard organization admin cannot reach usable advanced AI or enterprise training controls.
- Advanced personal, advanced employee, advanced organization admin, and content admin retain discoverable approved entries.
- `super_admin` workspace switching cannot present a login-like failure when the real issue is missing organization context.
- Organization admin pages do not imply employee import/mutation, organization tree mutation, or `org_auth` mutation authority.
- Content AI and organization AI do not imply direct formal publish.
- Learner AI result/history is not mixed into formal `practice`, `mock_exam`, `exam_report`, or `mistake_book`.
- `redeem_code` plaintext product UI exception remains role-limited and never appears in evidence.
- No branch changes `package.json`, lockfiles, `.env*`, DB schema, migrations, seed, fixtures, Provider execution/configuration, screenshots, traces, raw DOM, or private local design board files.
